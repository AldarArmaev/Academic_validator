Настоящий документ является техническим заданием на разработку системы автоматизированного анализа учебных работ (курсовых и дипломных проектов) на базе RAG-архитектуры и детерминированных правил. Документ составлен с учётом аналитической правки исходного ТЗ.

|**Параметр**|**Значение**|
| :- | :- |
|Наименование системы|Интеллектуальный Нормоконтролёр|
|Тип продукта|Веб-сервис (SaaS-MVP)|
|Целевая аудитория|Студенты и научные руководители вузов|

# **2. Архитектура системы**
Система разделена на два независимых вычислительных контура для обеспечения точности и экономии ресурсов. Ни один контур не должен выполнять функции другого.
## **2.1. Контур А — Детерминированный валидатор**
Отвечает исключительно за физические параметры документа. Работает без обращений к LLM-API.

|**Характеристика**|**Описание**|
| :- | :- |
|Входные данные|Файл .docx|
|Механизм|Извлечение XML-структуры (WordprocessingML)|
|Анализируемые теги|<w:rPr> — свойства прогона; <w:pPr> — свойства абзаца|
|Логика сравнения|Значения полей, кегля, интервала сравниваются с константами методички|
|Зависимость от LLM|Отсутствует|
## **2.2. Контур Б — Вероятностный семантический анализатор (RAG)**
Отвечает за смысл, стиль и верификацию цитат. Использует векторный поиск на основе эмбеддингов.

Косинусное сходство для поиска релевантных контекстов вычисляется по формуле:

similarity = cos(θ) = (A · B) / (‖A‖ · ‖B‖)

|**Характеристика**|**Описание**|
| :- | :- |
|Входные данные|Текст работы, PDF-источники, PDF-методичка|
|Механизм эмбеддингов|sentence-transformers (paraphrase-multilingual-MiniLM) — локально|
|Векторный поиск (MVP)|numpy cosine\_similarity, in-memory (без FAISS)|
|Векторный поиск (v2+)|FAISS или ChromaDB при объёме > 1000 документов|
|Ограничение домысливания|temperature = 0 + системный промпт "только из загруженных файлов"|
|⚠ temperature = 0 устраняет случайность, но не галлюцинации. Ограничение домысливания обеспечивается RAG-промптом: "Если информации нет в приложенных контекстах — укажи «не найдено»". Оба механизма обязательны.||

# **3. Функциональные требования**
## **3.1. Модуль препроцессинга и приватности (Privacy Shield)**
Согласно требованиям 152-ФЗ, персональные данные не передаются на сторонние API в открытом виде. Анонимизация обязательна до любого сетевого вызова.

|⚠ Замена ФИО через Natasha/SpaCy не обеспечивает полную защиту: название кафедры + фамилия научруководителя = деанонимизация по контексту. Для production рекомендуется on-premise LLM (Ollama + Mistral) или подписание DPA с LLM-провайдером.|
| :- |

Требования к анонимизации:

- Локальный скрипт на базе библиотеки Natasha (для русского языка) выполняет NER
- Замене подлежат: ФИО, названия кафедр и организаций, адреса электронной почты, номера телефонов
- Формат токена-заглушки: [ENTITY\_TYPE], например [STUDENT\_NAME], [DEPARTMENT]
- Анонимизация запускается до передачи текста в LLM-API

Парсинг и структурирование:

- Конвертация DOCX в структурированный Markdown с сохранением иерархии заголовков (H1, H2, H3)
- Детектирование глав по стилю абзаца: w:pStyle = Heading1 / Heading2 — используется для Map-Reduce
- Таблицы и рисунки: контент игнорируется, проверяется только наличие подписи (см. п. 3.2)
- Формулы OMML: для MVP — конвертация в LaTeX-нотацию через mathml2latex; при ошибке конвертации — игнорирование содержимого с проверкой наличия подписи
## **3.2. Модуль проверки оформления (Контур А)**
Все константы хранятся в файле university\_rules.json. Методичка вуза имеет приоритет над ГОСТом.

|**Параметр**|**Требуемое значение**|**XML-тег/атрибут**|
| :- | :- | :- |
|Левое поле|30 мм (1701 DXA)|w:pgMar w:left|
|Правое поле|10 мм (567 DXA)|w:pgMar w:right|
|Верхнее поле|20 мм (1134 DXA)|w:pgMar w:top|
|Нижнее поле|20 мм (1134 DXA)|w:pgMar w:bottom|
|Шрифт (гарнитура)|Times New Roman|w:rFonts w:ascii|
|Кегль|14 пт (28 в half-points)|w:sz|
|Межстрочный интервал|1\.5 (360 twips)|w:spacing w:line|
|Отступ первой строки|1\.25 см (720 DXA)|w:ind w:firstLine|
|Выравнивание|По ширине (both)|w:jc|

Дополнительные проверки структуры документа:

- Наличие обязательных разделов: Введение, Основная часть (не менее одной главы), Заключение, Список литературы
- Нумерация страниц: проверка наличия поля PageNumber в колонтитуле
- Таблицы: наличие подписи "Таблица N" до таблицы
- Рисунки: наличие подписи "Рисунок N" после рисунка
- Формулы: наличие идентификатора в скобках после формулы, например (1.1)
## **3.3. Модуль семантического анализа (Контур Б)**
### **3.3.1. Проверка научного стиля**
- Поиск местоимений первого лица: «я», «мы», «мне», «нам» и производных
- Рекомендация: замена на пассивный залог («автором изучено», «в работе рассмотрено»)
- Поиск разговорных оборотов через список стоп-слов (конфигурируемый)
- Система указывает фрагмент и формулирует рекомендацию — не переписывает текст за студента
### **3.3.2. Верификация ссылок на источники (Cross-referencing)**
- Извлечение списка литературы из раздела «Список литературы»
- Сопоставление внутритекстовых ссылок формата [N, с. X] с номером в списке
- RAG-запрос к PDF-файлу источника №N: присутствует ли указанная информация
- Результат: verified (найдено), not\_verified (не найдено в загруженном PDF), source\_missing (PDF не загружен)
### **3.3.3. Логический аудит глав**
- Для каждой главы: сопоставление содержания с заголовком через LLM-запрос
- Контекст запроса: текст главы + соответствующий раздел методички как системный промпт
- Стратегия: Map-Reduce — каждая глава анализируется отдельно, результаты агрегируются

# **4. Алгоритм работы (Pipeline)**
Полный цикл проверки состоит из пяти последовательных фаз. Контуры А и Б выполняются параллельно после фазы препроцессинга.

|**Фаза**|**Описание**|**Технологии**|
| :- | :- | :- |
|1\. Загрузка|Приём DOCX работы, PDF методички, PDF источников (до 5 файлов, ≤20 стр. работы)|FastAPI, multipart/form-data|
|2\. Препроцессинг|XML-парсинг DOCX, NER-анонимизация, разбивка PDF на чанки по 1000 токенов с overlap 10%|python-docx, Natasha, pdfplumber|
|3A. Контур А|Детерминированная проверка форматирования по константам university\_rules.json|python-docx, XML-анализ|
|3Б. Контур Б|Построение эмбеддингов, RAG-поиск, LLM Map-Reduce по главам|sentence-transformers, numpy, LLM API|
|4\. Агрегация|Объединение результатов Контуров А и Б в JSON-отчёт|Pydantic-схема|
|5\. Отдача|Передача JSON на фронтенд, удаление временных файлов и индексов|FastAPI, contextlib.ExitStack|

|⚠ Эмбеддинги PDF-источников кэшируются in-memory в рамках одной сессии. При повторном обращении к тем же источникам перестройка индекса не выполняется. Это критично для соблюдения лимита 5 минут.|
| :- |

# **6. Нефункциональные требования**

|**Требование**|**Описание**|
| :- | :- |
|Производительность|Полный цикл проверки — не более 5 минут. Эмбеддинги кэшируются в рамках сессии.|
|Прозрачность|Каждое замечание сопровождается цитатой из методички или указанием источника.|
|Приватность|Анонимизация через Natasha до передачи в API. Все временные файлы и векторные индексы удаляются по завершении сессии (contextlib.ExitStack).|
|Юридическая чистота|Авторские материалы (методички, книги) не сохраняются после сессии согласно ГК РФ ч. 4.|
|Масштабируемость MVP|Лимиты: 20 страниц работы, 5 PDF-источников, 1 PDF-методичка.|
|Интерфейс|Веб-сервис: Frontend — React, Backend — FastAPI.|
|Этика|Система не генерирует текст за студента. Роль — критический ментор.|

# **8. План реализации MVP (4 спринта, 7 недель)**
## **Спринт 1 (2 нед.) — Контур А, скелет проекта**
- python-docx: парсинг полей, шрифта, межстрочника, отступов через XML-теги
- Конфиг university\_rules.json с константами форматирования
- FastAPI endpoint POST /validate/format — возвращает список нарушений
- Определение JSON-схемы отчёта (Pydantic) — фиксируется до начала разработки фронтенда
- Покрытие unit-тестами: минимум 5 тестовых DOCX с заранее известными нарушениями
## **Спринт 2 (2 нед.) — Контур Б, RAG**
- pdfplumber: извлечение текста из PDF, разбивка на чанки 1000 токенов / overlap 10%
- sentence-transformers: построение эмбеддингов локально
- numpy cosine\_similarity: in-memory векторный поиск
- Map-Reduce: детектирование глав по w:pStyle = Heading1, каждая глава — отдельный LLM-запрос
- FastAPI endpoint POST /validate/semantic
## **Спринт 3 (1 нед.) — Privacy + интеграция**
- Natasha NER + замена токенами перед уходом в LLM API
- Очистка временных файлов и индексов через contextlib.ExitStack
- Сборка Контуров А и Б в единый pipeline, endpoint POST /validate
- Интеграционные тесты сквозного pipeline
## **Спринт 4 (2 нед.) — Frontend + бенчмарк**
- React: загрузчик файлов, прогресс-бар, рендер JSON-отчёта с подсветкой фрагментов в тексте
- «Золотой сет»: 5 работ с заранее размеченными ошибками, автоматический прогон
- Устранение дефектов по результатам бенчмарка
- Документирование API (OpenAPI/Swagger)

# **9. Критерии приёмки (Benchmark)**
Система считается работоспособной, если на тестовом наборе («Золотой сет» — не менее 5 размеченных работ):

|**Метрика**|**Порог приёмки**|
| :- | :- |
|Обнаружение отклонений в оформлении (шрифт, поля, интервалы)|≥ 95%|
|Обнаружение стилистических ошибок (местоимения, разговорные обороты)|≥ 80%|
|Корректная верификация связи текста с источником|≥ 70%|
|Ложноположительные срабатывания Контура А|≤ 5%|
|Время полного цикла проверки (20 стр. + 3 PDF-источника)|≤ 5 мин.|

|⚠ Порог 70% для RAG-верификации цитат отражает текущий уровень технологии. Повышение до 85% возможно в v2 через fine-tuning эмбеддинговой модели на академических текстах.|
